04 — Core Components
Every Keyword Explained Simply
What it is · Real-world analogy · How it is used in Stacy — grouped by layer
🤖 AI / Intelligence Layer
Core AI
NLP — Natural Language Processing
Enables computers to understand human language. "I lost my card" → computer understands: intent = report_lost_card. You don't need exact keywords.
🎯 Analogy: A person who understands what you mean, not just the exact words you use.
→ In Stacy: Every user message is processed through NLP inside the Morpheus engine to extract meaning and intent.
Core AI
Intent Recognition
Classifies a message into a category (intent). Examples: faq_minimum_balance, report_fraud, connect_live_agent. The intent determines what the bot does next.
🎯 Analogy: A receptionist who listens to you and decides which department to direct you to.
→ In Stacy: Morpheus detects intent → routes to FAQ answer, banking API, or Live Bank handoff.
AI Engine
Morpheus — AI Admin & App Server
Central brain of the chatbot. Receives all user messages, runs NLP, checks Redis cache, orchestrates the whole flow end-to-end.
🎯 Analogy: The call centre manager — routes every call to the right person and keeps records.
→ In Stacy: Receives messages → NLP → checks Redis → calls Trinity if cache miss → manages sessions → calls banking APIs.
AI Engine
Trinity AI Engine
Dedicated AI inference server. Only called when Redis cache has no answer (cache miss). Kept separate from Morpheus so config changes don't affect AI processing.
🎯 Analogy: The specialist expert the manager calls in when the standard answer isn't in the manual.
→ In Stacy: Deep NLP + ElasticSearch + Neo4j knowledge graph → generates the response.
Knowledge Graph
Neo4j — Knowledge Graph DB
Graph database — stores concepts and their relationships (not rows/columns like SQL). Ideal for "what topics relate to what intents?"
🎯 Analogy: A mind map — every concept is connected to related concepts.
→ In Stacy: Trinity uses Neo4j for intent mapping. ⚠️ Must reload after ES rollback or AI gives wrong answers.
⚡ Cache & Search Layer
Cache
Redis Cache
In-memory key-value store. Data lives in RAM (not disk) = microsecond lookups. Cache TTL = 30 minutes. Redis Sentinel provides auto-failover if master goes down.
🎯 Analogy: Post-it note on your desk. You read it instantly. No need to walk to the filing cabinet.
→ In Stacy: Morpheus checks Redis before every AI call. HIT = respond in ~50ms. MISS = forward to Trinity.
Search Engine
ElasticSearch
Distributed search engine — fast full-text search across large datasets. Cluster: 1 master + 2 data nodes. In DR failover, must point to DR ES instance.
🎯 Analogy: Google for your internal knowledge base — finds the most relevant answer from thousands of FAQs.
→ In Stacy: Trinity queries ElasticSearch to fetch and rank the best answer to the user's question.
🗄️ Storage Layer
Relational Database
Oracle DB
Enterprise relational database. Stores structured data in tables with SQL queries. ACID-compliant: Atomic, Consistent, Isolated, Durable.
🎯 Analogy: The official records room — every customer's file is precisely stored and instantly queryable.
→ In Stacy: Customer profiles, account data, transaction history, audit logs, chat transcripts (structured).
File Storage
NAS — Network Attached Storage
Centralized file storage over a network (NFS/SMB/CIFS). Stores files/blobs — NOT queryable with SQL. Very different from Oracle DB.
🎯 Analogy: Company shared network drive (like a mapped drive on Windows) — store documents, not databases.
→ In Stacy: AI config files, raw log files, chat transcripts as files, media, backups. Admin reads config at startup.
🔒 Security & Network Layer
Edge Security
Akamai BPM
CDN + Bot Protection Manager. Sits at the internet edge — filters traffic before it reaches any server. Blocks DDoS attacks and automated bot traffic.
🎯 Analogy: Bouncer outside the building — stops trouble before it walks through the door.
→ In Stacy: First line of defence. Protects the Prelogin API from bots and attack traffic.
API Gateway
Axway API Gateway
Enterprise API gateway for pre-login (MVP1). Authenticates requests, enforces rate limits (100/min), validates request format, logs everything for audit.
🎯 Analogy: Security desk at a corporate building — checks your ID, logs entry, controls which floors you can access.
→ In Stacy: Every pre-login API call passes through Axway. Kong IAG is used for post-login (MVP2+).
Load Balancer
F5 Load Balancer (ADC)
Application Delivery Controller. Distributes traffic across multiple app servers using Round Robin. Does SSL offloading (removes HTTPS processing from app servers). Health monitoring.
🎯 Analogy: Traffic policeman at a busy intersection — directs cars to different lanes so no single lane jams.
→ In Stacy: Sits in front of app cluster. Auto-removes unhealthy nodes. Industry standard in banking.
Network Zones
DMZ1 & DMZ2
DMZ1 (Outer): First boundary after internet. Hosts web server, F5. Publicly reachable — minimal services only.
DMZ2 (Inner): Between DMZ1 and internal network. Hosts app server, API gateway.
🎯 Analogy: Reception area (DMZ1) → Badge-restricted area (DMZ2) → Secure office floor (Internal Network).
→ In Stacy: Internal DB (Oracle, AI engine, ICM) is never directly reachable from the internet. Defense in depth.

07 — Interview Q&A
Top 10 Interview Questions
Click any question to expand the answer
Q1: Why is Redis used instead of querying Oracle DB every time? +
Oracle DB is disk-based — every query involves I/O, SQL parsing, and row retrieval = 50–500ms per query. Redis is in-memory = microseconds. The same FAQ is asked hundreds of times a day. The first user triggers Trinity AI (slow path). The next 500 users hit Redis and get the answer instantly.

The 30-minute TTL ensures answers don't become too stale.

Analogy: Post-it note on desk (Redis = instant) vs going to the filing cabinet every single time (Oracle DB = slow).
Q2: Why was Availability chosen over Consistency? (CAP Theorem) +
CAP Theorem states a distributed system can only guarantee 2 of: Consistency, Availability, Partition Tolerance. For Stacy:

  • Returning a slightly stale FAQ answer (cached 25 min ago) = acceptable
  • But if chatbot is DOWN = customers call hotline = defeats the entire purpose

Cost of unavailability >> cost of slight inconsistency. Redis with 30-min TTL directly implements this decision.
Q3: Why two API Gateways — Axway and Kong? +
They serve different trust zones:

  • Axway — MVP1 pre-login (unauthenticated, public-facing). Enterprise-grade, strong audit trail, proven for public APIs.
  • Kong IAG — MVP2+ post-login (authenticated, internal). Flexible routing to ICM and CSC banking APIs.

Separating gateways for public vs authenticated traffic is a security best practice — different rate limits, policies, and token types for each zone.
Q4: How does post-login stay secure — no long-lived credentials? +
SC Mobile sends a short-lived access token (not a password) → Morpheus uses it to call the Customer API to prove identity.

For Live Bank handoff: CASAS issues a single-use SSO token → passed to Live Bank → validated → expires.

The chatbot never stores passwords, PINs, or long-lived secrets. Even if the chatbot were compromised, an attacker would only find already-expired tokens.
Q5: How do you scale the system from 5,000 to 50,000 users? +
Strategy: Horizontal scaling at every layer — no single layer becomes the bottleneck.

  • App Servers: Add Node 4, 5... behind F5. Zero downtime — F5 auto-discovers new nodes.
  • Redis: Expand to cluster — multiple shards handling different key ranges.
  • Oracle DB: Add read replicas + connection pooling for high throughput.
  • Trinity AI: Deploy multiple instances behind a load balancer.
  • ElasticSearch: Add data nodes to the cluster.

The app layer is stateless — any node can handle any request — making horizontal scaling seamless.
Q6: What happens if Redis goes down? +
Redis Sentinel monitors the Redis master node. If it goes down:

  • Sentinel detects the failure within seconds
  • Automatically promotes a replica to become the new master
  • All clients redirect to the new master

During the brief failover window, cache miss requests fall through to Trinity AI — responses are slower (~1–2 sec) but the chatbot remains fully functional. This is by design — Availability > Consistency.
Q7: What is the difference between NAS and Oracle DB? +
NASOracle DB
File/blob storageRows and columns (SQL)
No SQL queriesFull SQL + ACID
Config files, logs, transcripts as files, mediaCustomer profiles, audit, transactions

Rule: If you need to query it with SQL → Oracle DB. If you just store and retrieve it as a file → NAS.
Q8: Why is the service startup order critical in DR? +
Each service depends on the previous one. Wrong order = services appear "up" but chatbot gives wrong responses.

Correct startup order:
  1. Supervisor (auto-starts Manager + Triniti)
  2. ElasticSearch (1 master + 2 data nodes)
  3. Redis Server → then Redis Sentinel
  4. Neo4j (knowledge graph)
  5. Morpheus + Admin + JBoss (app layer — last, depends on all above)
Q9: What is the Neo4j reload after ES rollback? Why is it critical? +
ElasticSearch and Neo4j are tightly coupled. When you roll back ES to a previous index, the content changes. If Neo4j is not reloaded, it still has the graph from the newer version — the relationships no longer match the ES content.

Result: Trinity AI processes a query, navigates Neo4j, queries ES with wrong keys → incorrect/empty chatbot responses.

Fix path: Login Admin → Manage AI → Knowledge Graph → Reload Graph

⚠️ This step is frequently missed in DR drills. Always include it in the runbook.
Q10: Why does the post-login use WebSocket instead of REST? +
REST: Client sends request → server responds → connection closes. Like sending a letter. Works fine for FAQ queries.

WebSocket: Connection stays open permanently. Both sides can send messages at any time. Like a phone call.

Post-login needs WebSocket because:
  • Live Bank agent chat requires real-time bidirectional communication
  • Server needs to push messages to client without being asked
  • REST would require constant polling (inefficient and slow)
